# Current Task

## Task ID
TASK-011

## Status
READY

## Priority
Medium

## Title
Развить личность Марины и осмысленное использование памяти в chat responses

## Goal
Сделать ответы Марины заметно более живыми и последовательными без подключения внешнего AI: вынести детерминированную personality/memory policy из набора простых keyword-веток в тестируемый backend service, который учитывает текущее эмоциональное состояние Марины и релевантные недавние воспоминания, сохраняя существующий API и безопасность.

## Instructions
1. Начать работу от актуального `main` после merge PR #9.
2. Прочитать файлы в порядке, указанном в `AGENTS.md` и `docs/CODEX_PROTOCOL.md`.
3. Создать отдельную ветку для TASK-011.
4. Проверить текущие `backend/app/game_services.py`, `backend/app/models.py`, `backend/app/schemas.py`, chat route и существующие service/HTTP/idempotency tests.
5. Сохранить полностью детерминированную локальную реализацию. Не подключать OpenAI, внешние LLM, сеть, embeddings, vector database или новые секреты.
6. Вынести personality/memory response logic в отдельный небольшой модуль или чётко изолированный service layer, чтобы `apply_chat_message()` отвечал за orchestration, persistence и state mutation, а не за длинный набор текстовых веток.
7. Добавить понятную классификацию пользовательского сообщения минимум для следующих intent-категорий:
   - любовь/нежность;
   - извинение/примирение;
   - грусть/усталость/поддержка;
   - совместное воспоминание;
   - вопрос;
   - нейтральный разговор.
8. Ответ должен учитывать текущее состояние `MarinaState` минимум по `mood`, `trust`, `love`, `calm` и `energy`. Низкие и высокие значения должны приводить к проверяемо различающемуся тону/варианту ответа, но без оскорблений, угроз, давления, манипуляций или эмоционального шантажа.
9. Улучшить memory selection:
   - не использовать безусловно только последнюю пользовательскую запись;
   - выбирать из ограниченного числа недавних `MarinaMemory` только пользовательские воспоминания;
   - не подставлять текущий текст как прошлое воспоминание;
   - не цитировать пустой, слишком короткий или явно нерелевантный контент;
   - ограничивать длину вставляемой памяти и не раскрывать служебные данные.
10. Для сообщений о прошлом (`помнишь`, `раньше`, `вчера` и аналогичные) использовать выбранную память, если она есть; при отсутствии памяти отвечать естественно без выдуманного факта.
11. Сохранить существующие response schema и HTTP contract `/api/v1/chat`: `reply`, `emotion`, `remembered`, `player`.
12. Поле `remembered` должно содержать фактически выбранную прошлую пользовательскую память либо `null`; запрещено возвращать выдуманную память.
13. Сохранить idempotency behavior: повтор одного и того же chat request с тем же ключом не должен создавать дополнительные memories или повторно менять состояние.
14. Сохранить безопасные границы изменения характеристик 0–100 и существующую выдачу опыта, если нет доказанной причины для минимальной корректировки. Не менять игровую экономику действий.
15. Не менять DB schema и Alembic revisions в рамках этой задачи. Использовать существующую таблицу `marina_memories`.
16. Добавить автоматические тесты минимум для:
   - intent classification;
   - различия тона при низком и высоком emotional state;
   - выбора подходящей прошлой пользовательской памяти из нескольких недавних записей;
   - игнорирования Marina/event memories при выборе пользовательского воспоминания;
   - естественного ответа без памяти и без выдуманного `remembered`;
   - ограничения длины цитируемой памяти;
   - сохранения chat state mutations, memory persistence и response schema;
   - отсутствия регрессии idempotent replay/conflict behavior.
17. Добавить HTTP-level или service-level regression test, подтверждающий, что текущий chat request не возвращается как собственное прошлое воспоминание.
18. Не выполнять широкий рефакторинг frontend. Frontend менять только при доказанной несовместимости, которой в норме быть не должно.
19. Обновить `docs/ARCHITECTURE.md`, `docs/PROJECT_STATE.md`, `docs/TECH_DEBT.md`, `docs/ROADMAP.md` и `docs/CHANGELOG.md` только по фактически выполненным изменениям.
20. Полностью заменить `docs/REPORT.md` отчётом TASK-011 согласно протоколу.
21. После завершения изменить статус задачи на `DONE`, сделать commit, push и открыть отдельный PR в `main`.

## Acceptance Criteria
- Personality/memory response policy изолирована и покрыта автоматическими тестами.
- Ответы различаются с учётом intent и текущих `mood`, `trust`, `love`, `calm`, `energy`.
- `remembered` содержит только реально сохранённую прошлую пользовательскую память либо `null`.

- Marina/event records не используются как пользовательская память.
- Ответ без найденной памяти не выдумывает прошлые события.
- API schema, Telegram auth, idempotency semantics и DB schema сохранены.
- Backend-тесты, frontend tests и frontend production build проходят.
