## Title
Backend production-readiness audit and targeted hardening

## Goal
Проверить фактическое состояние backend-кода Telegram Mini App «День Марины» и устранить подтверждённые критичные или высокоприоритетные проблемы, связанные с управлением схемой базы данных, CORS и идемпотентностью операций, изменяющих игровую экономику.

## Required Reading
Перед началом работы прочитать в указанном порядке:
1. `AGENTS.md`
2. `docs/CODEX_PROTOCOL.md`
3. `docs/AI_CONTEXT.md`
4. `docs/PROJECT_STATE.md`
5. `docs/TECH_DEBT.md`
6. `docs/TASK.md`

## Instructions
1. Провести аудит backend без предположений и зафиксировать фактическое поведение текущего кода.
2. Проверить, используется ли runtime `create_all()` как основной механизм управления схемой БД.
3. Если полноценные миграции отсутствуют, внедрить минимально корректную конфигурацию Alembic и создать базовую миграцию, совместимую с текущими моделями.
4. Проверить CORS-конфигурацию для production:
   - не допускать небезопасный wildcard вместе с credentials;
   - использовать явный список origin из переменных окружения;
   - сохранить удобную локальную разработку без ослабления production-настроек.
5. Найти endpoint-ы и сервисы, которые изменяют валюту, энергию, инвентарь, прогресс или иные экономические данные.
6. Проверить защиту таких операций от повторной отправки одного и того же запроса.
7. Реализовать идемпотентность только там, где проблема подтверждена и исправление можно сделать без ломки публичного API. Если требуется изменение API-контракта, подробно описать это в отчёте и не выполнять рискованный рефакторинг без необходимости.
8. Добавить или обновить автоматические тесты для каждого внесённого исправления.
9. Не делать несвязанный рефакторинг frontend или визуальной части приложения.
10. Не изменять секреты, production-данные и Railway-конфигурацию напрямую.
11. Не выполнять merge автоматически.

## Validation
Обязательно выполнить доступные проверки проекта, включая:
- backend unit/integration tests;
- проверку импорта и запуска FastAPI-приложения;
- проверку Alembic configuration и построения migration graph;
- lint/type checks, если они уже настроены в проекте.

Если часть проверок невозможно выполнить в окружении, указать точную причину и команды, которые нужно запустить вручную.

## Acceptance Criteria
- Фактическое состояние migrations, CORS и economy idempotency задокументировано.
- Подтверждённые high-severity проблемы исправлены либо чётко объяснено, почему безопасное исправление невозможно в рамках задачи.
- Для внесённых исправлений существуют тесты или воспроизводимые проверки.
- Существующие тесты не сломаны.
- `docs/REPORT.md` содержит список проверенных файлов, найденных проблем, изменений, тестов, рисков и оставшихся рекомендаций.
- `docs/PROJECT_STATE.md` обновлён по фактическому состоянию проекта.
- `docs/TECH_DEBT.md` обновлён: подтверждённые проблемы добавлены, исправленные удалены только при наличии доказательства.
- `docs/CHANGELOG.md` обновлён, если код был изменён.
- После завершения статус этой задачи изменён на `DONE`.
- Изменения закоммичены в отдельной ветке и оформлены в Pull Request, если среда позволяет.
