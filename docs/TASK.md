## Title
Добавить idempotency keys во frontend-запросы chat и actions

## Goal
Задействовать уже реализованную backend-защиту от повторной обработки экономически значимых запросов: frontend должен отправлять корректный `idempotency_key` для каждого пользовательского запроса к `POST /api/v1/chat` и `POST /api/v1/actions`, не меняя API backend, игровой баланс и пользовательское поведение.

## Instructions
1. Начать работу от актуального `main` после merge PR #4.
2. Прочитать файлы в порядке, указанном в `AGENTS.md` и `docs/CODEX_PROTOCOL.md`.
3. Создать отдельную ветку для TASK-006.
4. Проверить текущие frontend-вызовы `/api/v1/chat` и `/api/v1/actions`, Pydantic-схемы backend и существующие idempotency-тесты.
5. Добавить frontend-генерацию idempotency key для каждого нового намеренного запроса пользователя:
   - новый запрос пользователя получает новый ключ;
   - ключ должен быть строкой, совместимой с текущим backend-ограничением;
   - предпочтительно использовать `crypto.randomUUID()` с безопасным локальным fallback для окружений, где он недоступен.
6. Передавать поле `idempotency_key` в JSON payload запросов chat и actions.
7. Один и тот же логический запрос должен сохранять один ключ на время выполнения и возможного повторного вызова транспортного слоя; не генерировать новый ключ внутри низкоуровневой повторной отправки одного payload.
8. Не использовать постоянный ключ между разными сообщениями, действиями, пользователями или сессиями.
9. Не менять backend API, Pydantic-схемы, idempotency storage, fingerprint-логику, HTTP status codes и схему базы данных, если фактическая несовместимость не доказана.
10. Не менять тексты интерфейса, игровой баланс, начисления/списания, список действий, Telegram auth и визуальное поведение.
11. По возможности вынести построение mutation payload/ключа в небольшой проверяемый frontend helper, не выполняя широкий рефакторинг `App.tsx`.

Если её нет, не добавлять тяжёлый test framework только ради этой задачи; вместо этого выполнить production build и точно описать ручную/статическую проверку в отчёте.
